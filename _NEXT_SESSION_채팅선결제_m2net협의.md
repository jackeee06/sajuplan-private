  - ③ 잔액 기준 5분 전 — "⏰ 상담 종료 5분 남았습니다"
  - ④ 잔액 0 도달 — 자동 종료
- **상담사 미입장 시 차감 X** — 채팅·전화 모달 양쪽에 강조 표시

## 2. 현재 사주플랜 ↔ m2net 잔액 동기화 메커니즘

### 충전 시
1. 회원이 PG 로 사주플랜에 충전 → 사주플랜 `point` (paid_balance/free_balance) 증액
2. **m2net memb-mgrp API** 호출 → m2net 측 회원 잔액도 동기 증액
3. 양쪽 잔액 = 일치

### 채팅 중 (m2net 이 차감 주체)
- m2net 의 **wss 채팅 메시지 또는 통화 시간 단위로 m2net 측이 자체적으로 회원 잔액 차감**
- 사주플랜은 이 동안 차감 X (회원 잔액 변동 없음)
- 즉 **m2net = 실시간 차감 주체**, 사주플랜 = 미러러

### 종료 시 (사주플랜이 m2net 결과 미러링)
- m2net 이 END_CHAT push 발송
- 사주플랜 `settleChatRoomLocal()` 호출
- 핵심 로직 (`api/src/pg-callbacks/m2net-push.service.ts:538~570`):
  ```ts
  const sajumoonBalance = (사주플랜 측 회원 잔액)
  const m2netBalance = m2net.getMemberByMembid(membid) 응답
  const diff = sajumoonBalance - m2netBalance
  if (diff > 0) {
    // m2net 이 차감한 만큼 사주플랜도 차감 (미러링)
    deductMemberPoint(memberId, diff, ...)
  } else {
    // m2net 잔액 >= sajumoon → 차감 skip
    return { settled: false }
  }
  ```

## 3. 사장님 시나리오 추적 — 15분 선결제 + 11분 사용

**전제**
- 회원 잔액 양쪽 100,000 코인 (동기 상태)
- 단가 1,500코인 / 30초 (m2net 도 동일)

### Phase 0: 시작 전
| | 사주플랜 | m2net |
|---|---|---|
| 잔액 | 100,000 | 100,000 |
| 상태 | 동기 ✅ | 동기 ✅ |

### Phase 1: 회원이 15분 선택 → 사주플랜만 선결제 차감
- 15분 × 60초 / 30초 × 1,500 = **45,000 코인 차감**
- 사주플랜만 차감, m2net 은 변동 X

| | 사주플랜 | m2net |
|---|---|---|
| 잔액 | 55,000 | 100,000 |
| 상태 | ⚠ 불일치 시작 | |

### Phase 2: 11분 채팅 사용 → m2net 이 실시간 자체 차감
- 11분 × 60초 / 30초 × 1,500 = **33,000 코인 m2net 측에서 차감**
- 사주플랜은 변동 X

| | 사주플랜 | m2net |
|---|---|---|
| 잔액 | 55,000 | 67,000 |
| 상태 | ⚠ 12,000 불일치 | |

### Phase 3: 종료 → settleChatRoomLocal 호출
- diff = sajumoonBalance - m2netBalance = 55,000 - 67,000 = **-12,000 (음수)**
- 현재 코드: diff <= 0 이면 **차감 skip + 로그 "m2net 잔액 >= sajumoon → 차감 skip"**

| | 사주플랜 | m2net |
|---|---|---|
| 최종 잔액 | 55,000 | 67,000 |
| 불일치 | **12,000 코인** | |

### 결과 — 회원이 보는 상태
- 사주플랜 앱: 55,000 코인 (선결제분 45,000 차감)
- m2net 어드민 / 다른 결제 시점: 67,000 코인 (실사용분 33,000 차감)
- **회원이 어드민에서 보면 두 숫자가 다름** → 혼란 + 클레임
- 차이 12,000 = 사장님이 원한 "선결제 약속" 분이지만, m2net 잔액엔 반영 안 됨

### 동일 패턴 사고 이력
- **2026-05-22 PG-m2net 이중 적립 사고**가 동일 패턴 (양쪽 잔액 불일치) → 메모리 `project_pg_m2net_double_fill.md` 참조
- 안전망 적용했지만 근본 원인은 양쪽 잔액 동기화 메커니즘 부재

## 4. 해결 옵션 3가지

### A안 — m2net 측 단가를 0 또는 매우 낮게 설정 (가장 깔끔)

- m2net 시스템은 차감 안 함 + 사용 시간 측정만
- 사주플랜이 자체 차감 (선결제 또는 실사용 자유롭게 선택 가능)

**구현**
- passcall.co.kr (m2net 운영사) 협의 — "m2net 단가를 0 으로 설정해 줄 수 있는지" 또는 "특정 상담사/회원군은 m2net 차감 X 설정 가능한지"
- 사주플랜은 자체 use_seconds + 선결제분 기반 자유롭게 정산

**장점**
- 사장님 원래 정책 100% 구현 가능 (선결제 + 환불 X + 자동 연장)
- 양쪽 잔액 = 항상 일치 (m2net 측 차감 0)

**단점**
- m2net 운영사 협의 필요 — 기술적 가능 여부 + 정책 가능 여부 미확인
- 협의 시간 소요

**확인 사항 (m2net 에 질문)**
1. m2net 측 단가를 0 으로 설정 가능한가?
2. 특정 회원 또는 상담사 단위로 차감 X 설정 가능한가?
3. 시간 측정 기능만 사용하고 차감은 사주플랜이 자체 처리하는 운영 모델 가능한가?

**장점**

**적용 가능성**
- m2net 협의 무관, **지금 바로 가능**
- 이번 세션 작업 (ConsultModal UI + 자동 연장 + 알림 4단계 + STAY 가드) 그대로 완성

### C안 — m2net 통신 추가 + 선결제 강제 (위험)

**개념**
- 시작 시점에 사주플랜 + m2net **양쪽 동기 차감** (memb-mgrp API)

**위험**

- 시작/종료/정산 모든 흐름의 멱등성 검증 필요
- 환불 처리 흐름까지 손봐야 (시스템 장애 환불)

**장점**
- 사장님 원래 정책 100% 구현 + m2net 운영사 협의 불필요

**단점**
