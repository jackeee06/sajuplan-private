> **작성일**: 2026-05-23
> **목적**: 1주일 후 2단계 작업을 위한 완전한 인계 문서 + 사장님 보관용
> **위험도**: 1단계 안전망 적용 후 → 잔액 사고 위험 0 (자동 정정 작동)

## 🎯 진짜 원인 — 100% 확정

### 사고 발생 흐름
```
1. 회원이 사주플랜에서 가상계좌 입금
2. AG9(PG) 가 m2net 에 직접 fill 호출 (사주플랜 미경유, 충전방법 "P")
   → m2net 측 잔액 +코인량
3. AG9 가 사주플랜에도 콜백 push
4. 사주플랜의 charge.service.ts handleVbankCallback 이 콜백 처리
   → 사주플랜 자체적으로 m2net.addMemberCoin 호출 (충전방법 "I")
   → m2net 측 잔액 +코인량 (2번째 누적)
5. 결과: m2net 측에 같은 결제 +60,000 누적 (실제는 +30,000 만 정상)
```

### 사주플랜의 자동 정정 (이미 작동 중)
- `auth.service.ts:142 syncM2netBalanceForMember`
- 사용자 로그인 또는 마이페이지 진입 시 자동 호출
- `getMemberByMembid` 가 m2net 측에서 404 응답
- fallback 으로 `updateMember(amt: me.point)` 호출 (사주플랜 절대 잔액으로 m2net amt overwrite)

### 추가된 함수 — `correctM2netDoubleFill`
```typescript
/**
 * [2026-05-23] PG-m2net 직통 이중 적립 자동 정정.
 *
 * 사고 시나리오:
 *   - AG9(PG) 가 결제 완료 시 m2net 에 +coin 직통 fill (사주플랜 미경유)
 *   - 사주플랜도 addMemberCoin(+coin) 호출
 *   → 같은 결제가 m2net 측에 2배 누적됨
 *
 * 정정 방법:
 *   결제 완료 후 2초 지연(직통 fill 완료 대기) → 사주플랜 잔액으로 m2net amt overwrite.
 *   updateMember(amt) 는 매뉴얼 §3.4 의 절대값 셋팅 (overwrite). 동작 검증됨.
 *
 * 비동기 백그라운드 호출 — 응답 지연 방지. 실패 시 syncM2netBalanceForMember 가
 * 다음 로그인 시 다시 정정 시도하므로 결제 자체는 영향 없음.
 */
private async correctM2netDoubleFill(
  memberId: number,
  m2netMembid: string,
  paymentId: number,
): Promise<void> {
  try {
    await new Promise((resolve) => setTimeout(resolve, 2000));
    const ptRows = await this.sql<{
      free_balance: number;
      paid_balance: number;
    }[]>`
      SELECT free_balance, paid_balance FROM point WHERE member_id = ${memberId} LIMIT 1
    `;
    if (ptRows.length === 0) return;
    const sajumoonAmt =
      Number(ptRows[0].free_balance) + Number(ptRows[0].paid_balance);
    const r = await this.m2net.updateMember(m2netMembid, { amt: sajumoonAmt });
    if (r.ok) {
      this.logger.log(
        `[postPaymentSync] member=${memberId} m2net=${m2netMembid} payment=${paymentId} amt=${sajumoonAmt} → overwrite OK`,
      );
    } else {
      this.logger.warn(
        `[postPaymentSync] overwrite 실패 member=${memberId} payment=${paymentId}: ${r.error}`,
      );
      void this.opsAlert.send(
        'M2NET 잔액 정정 실패(post-payment sync)',
        `payment.id=${paymentId} member=${memberId} m2net_membid=${m2netMembid}\n` +
        `사주플랜 잔액=${sajumoonAmt} 로 m2net overwrite 시도했으나 실패.\n` +
        `m2net error: ${r.error ?? 'unknown'}\n` +
        `→ 사용자 재로그인 시 syncM2netBalanceForMember 가 다시 정정. 모니터링 필요.`,
      );
    }
  } catch (e) {
    this.logger.warn(
      `[postPaymentSync] 예외 member=${memberId} payment=${paymentId}: ${e instanceof Error ? e.message : String(e)}`,
    );
  }
}
```

## 📊 1주일 모니터링 체크리스트

### 매일 확인할 것

#### 1) m2net 어드민의 충전이력
- 모든 결제건에 **P (PG직통) 와 I (API) 두 줄이 함께** 기록되는지
- 만약 P 만 있고 I 가 없는 결제 → 사주플랜 코드가 호출 안 됨 (사주플랜 버그)
- 만약 I 만 있고 P 가 없는 결제 → AG9 직통 실패 (m2net 측 이슈) → 2단계 작업 보류!

#### 2) m2net 어드민의 현재잔액 = 사주플랜 잔액 일치
사장님이 직접 비교:
```
m2net 어드민 → 회원원부 → 해당 회원 현재잔액
↓ 일치 ↓
사주플랜 사용자 마이페이지 → 소비포인트
```

#### 3) OpsAlert 알림 확인
`M2NET 잔액 정정 실패(post-payment sync)` 메시지가 BizM 카톡으로 오는지 모니터링.
- 안 오면 → 정정 100% 성공 → 2단계 안전
- 오면 → m2net 측 정정 실패 케이스 발견 → 2단계 보류 + 원인 분석

### SQL 점검 (필요 시)
```sql
-- 최근 7일 모든 결제 + m2net 적립 상태
SELECT id, member_id, mb_1, pay_method, amount, coin_amount,
       status, m2net_status, m2net_retry_count, created_at, updated_at
  FROM payment
 WHERE created_at >= NOW() - INTERVAL '7 days'
   AND status = 'completed'
 ORDER BY id DESC;
```

```sql
-- 모든 회원의 사주플랜 잔액 (분석 비교용)
SELECT m.id, m.mb_id, m.name, m.m2net_membid,
       p.free_balance, p.paid_balance,
       (p.free_balance + p.paid_balance) AS sajumoon_total
  FROM member m JOIN point p ON p.member_id = m.id
 WHERE m.m2net_membid IS NOT NULL
 ORDER BY m.id;
```

## ⚠️ 위험 시나리오 & 대응

### 1단계 적용 상태 (현재) — 안전
- 이중 적립은 일어남 (m2net 측 +60k)
- 결제 +2초 후 사주플랜이 m2net 잔액을 사주플랜 잔액으로 overwrite → 정확히 정정
- **사용자 영향 0, 회사 부담 0** (잔액 부풀림이 즉시 정정됨)

### 1단계 안전망 실패 케이스 (드물게)
- `correctM2netDoubleFill` 호출 자체가 실패할 경우
- m2net 잔액이 부풀려진 채로 유지
- → 다음 로그인 시 `syncM2netBalanceForMember` 가 또 정정 시도 (이중 안전망)
- → 사용자가 끝까지 로그인 안 하면 → 통화 시 m2net 차감되며 사주플랜 차감보다 늦어짐 → 사주플랜은 잔액 0 인데 m2net 양수 → 회원 통화 가능 (회사 부담)
- 이런 케이스 발견 시 OpsAlert 가 발송됨

### 2단계 적용 후 위험
- AG9 직통 fill 이 어쩌다 실패한 결제 1건이 있다면
- 사주플랜 코드도 m2net 호출 안 하니까 m2net 적립 누락
- 회원이 통화 불가 → 사용자 불만 + 회사 신뢰도 손상
- **대응**: 즉시 1단계로 롤백 (`charge.service.ts` 의 `addMemberCoin` 호출 복원)

## 🤔 매뉴얼-실제 동작 정리

### m2net API 매뉴얼 vs 실제

| Endpoint | 매뉴얼 명시 | 실제 동작 | 비고 |
|---|---|---|---|
| `PUT /memb-mgr/{membid}` body `{amt}` | overwrite | **overwrite ✅ 확정** | 1단계 적용으로 검증됨 |
| `PUT /memb-mgr/{membid}/fill` body `{amt}` | +/- 델타 | (가정상 정상) | 사주플랜이 addMemberCoin 으로 사용 |
| `GET /memb-mgrp/{cpid}/{json}` | 회원 조회 | **404 ❌** | 매뉴얼-서버 불일치. 사주플랜 syncM2net 가 fallback 사용 |

### `sajumoon01` 정체
- m2net 어드민의 변경ID 컬럼에 자주 나타남
- 추정: 사주플랜이 m2net API 를 호출할 때 m2net 측 인증키로 식별되는 호출자명
- 즉 **사람이 m2net 어드민에서 손댄 게 아니라 사주플랜 시스템이 호출한 것**
