# 권한 가드 하드닝 — JWT sub 타입 함정 (2026-06-11)

## 한 줄로 답하면
**로그인 토큰 안의 "내 번호(sub)"가 글자(문자열)로 들어와서, 숫자로 된 회원 번호와 그냥 비교하면 "다른 사람"으로 잘못 판단될 수 있었다. 회원·관리자 양쪽 로그인 검문소(가드)에서 들어오자마자 숫자로 바꿔주도록 막아뒀다.**

---

## 무슨 문제였나 (쉬운 말로)

로그인하면 서버는 "이 사람은 141번 회원"이라는 출입증(JWT 토큰)을 준다.
그런데 이 출입증 안의 번호가 **글자 "141"** 로 들어오는데, 데이터베이스의 회원 번호는 **숫자 141** 이다.

컴퓨터에게 글자 `"141"` 과 숫자 `141` 은 "같냐?"고 물으면 **"다르다"** 고 답한다.
그래서 "본인인지" 또는 "이 데이터가 내 것인지" 확인하는 코드에서 이 둘을 직접 비교하면,
**본인인데도 남으로 처리되거나, 반대로 차단해야 할 걸 통과**시키는 일이 생길 수 있다.

실제로 2026-06-11 점검 때 회원 쪽에서 4곳이 이 함정에 걸려 있었고(본인이 자기에게 문의/상담/단골 등록되는 일이 실제로 발생), 즉시 고쳤다.

---

## 어떻게 막았나

검문소(가드)에서 출입증을 받자마자 번호를 **숫자로 변환**해서 통과시킨다.
- 회원 가드(`user-auth.guard`): 2026-06-11 적용 완료
- 관리자 가드(`admin-auth.guard`): **2026-06-11 추가 적용** — 관리자 쪽은 번호를 "누가 처리했는지" 기록용으로만 쓰고 있어 당장 새는 곳은 없었지만, 앞으로 권한·소유 비교 코드를 새로 짤 때 같은 실수를 원천 차단하려고 똑같이 막아뒀다.

이제 번호가 깨진(숫자로 못 바꾸는) 출입증은 **로그인 거부**된다.

## 이건 정상인가요? 에러인가요?

| 상황 | 판단 |
|---|---|
| 비로그인으로 회원/관리자 기능 호출 → 401 | ✅ 정상 (막힘) |
| 회원 계정으로 상담사 전용(선지급/단가) 호출 → 403 | ✅ 정상 (역할 차단) |
| 회원/상담사 토큰으로 관리자 정산·포인트 호출 → 401 | ✅ 정상 (관리자 가드 차단) |
| 본인에게 문의/상담/단골 → 4xx | ✅ 정상 (2026-06-11 수정 후) |
| 가용 초과 선지급 신청 → 400 | ✅ 정상 (한도 차단) |

---

## 점검은 어떻게 했나 (적대적 검증)

"깨뜨리려" 일부러 잘못된 요청을 던져 **전부 막히는지** 확인하는 방식(축2).
실제 돈 이동이나 데이터 생성은 0건 — 전부 "차단" 케이스만 테스트해 운영 데이터 오염이 없다.

- 상담사 기능: `e2e/tests/57-counselor-edge.spec.ts` (17건 — 음수/0/가용초과/상한초과/남의 데이터/역할 차단)
- 관리자 돈 라우트: `e2e/tests/58-admin-guard-edge.spec.ts` (12건 — 비로그인·회원·상담사 토큰 전부 401)
