# 사고 발생 시 어떻게 대응해요?

## 한 줄로 답하면
**critical / warning / info 3단계 분류. critical 은 즉시 대응 (분 단위), warning 은 시간 단위, info 는 일·주 단위.**

## 사고 분류

### 🔴 critical (즉시 대응)
- 사주플랜 사이트 다운 (HTTP 5xx)
- 데이터베이스 연결 사고
- m2net 응답 모두 실패 (통화·채팅 불가)
- PG (AG9) 응답 모두 실패 (결제 불가)
- 회원 다수 동시 클레임

### 🟡 warning (시간 단위)
- m2net 잔액 동기화 누락
- 알림톡 발송 실패 다수
- 채팅 STAY 무한 대기 (cron 정지)
- 정산 cron 결과 이상

### 🟢 info (일·주 단위)
- 일일 요약 알림 미수신
- 일부 회원 환불 클레임
- 부재중 상담사 다수

## 빠른 진단

### 사주플랜 다운 의심
1. 사이드바 → 대시보드 접속 — 안 되면 시스템 사고
2. https://sajuplan.com 접속 — 안 되면 사이트 다운
3. 사장님 + 개발자에게 즉시 알림

### 결제 사고 의심
1. 사이드바 → 매출현황 → 결제 내역
2. 최근 결제 status='paid' 잘 들어오는지
3. 실패 다수면 AG9 측 사고 의심

### m2net 사고 의심
1. 회원이 통화·채팅 안 됨 다수 클레임
2. 사이드바 → 매출현황 → 사용(상담) 내역 → reason 컬럼 확인
3. NO_ANSWER_CSR 등 다수면 m2net 측 사고 의심

## 절대 하지 말 것

- ❌ DB 직접 회원 삭제 / TRUNCATE
- ❌ 환경 치환 (`__SAJUMOON_ENV__`) 안 한 채 배포
- ❌ 운영 중 외부 서비스 키 임의 수정
- ❌ 사고 원인 분석 없이 결제·정산 데이터 직접 수정

## 사고 보고 흐름

1. 사고 발견 → 즉시 사장님 (운영자) 에게 알림
2. 가능하면 1차 조치 시도 (서비스 복구 우선)
3. 원인 분석 (운영 바이블 + 코드)
4. 학습 — 매뉴얼·메모리에 박제

## 운영자가 확인하는 위치

| 확인 항목 | 메뉴 위치 |
|---|---|
| 대시보드 (운영 상태) | 사이드바 → 대시보드 |
| 결제 내역 | 사이드바 → 매출현황 → 결제 내역 |
| 환불 이력 | 사이드바 → 매출현황 → 환불 이력 |
| 사용 내역 | 사이드바 → 매출현황 → 사용(상담) 내역 |
| 운영 KPI | 사이드바 → 매출현황 → 운영 KPI |
