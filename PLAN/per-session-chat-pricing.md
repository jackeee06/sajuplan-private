# 건당 채팅상담 과금 정책 — 설계 개념

> **상태**: 개념 정리 (m2net 협의 진행 중, 코드 미시작)
> **최종 업데이트**: 2026-05-25
> **계기**: 사장님이 m2net 에 "채팅 1건당 15분 N원" 가능 여부 문의 → m2net 답변 (foxtail@mtonet.co.kr, 2026-05-24)

---

## 1. 배경

현재 사주플랜 채팅 상담 모델:
- **종량제**: 분당(30초당) 차감. 상담 길어질수록 사용자 코인 계속 줄어듦.
- m2net 단가 등록: 상담사별 개별요율 (예: 1,000원/30초)
- 사용자 입장: "내 코인이 줄어드는 게 부담"
- 사장님 의도: **건당(예: 15분 정액) 정책으로 전환**해서 사용자 부담 줄이기 + 코인 잔액 자각 줄이기

---

## 2. m2net 답변 핵심 (2026-05-24)

### 결론
- **건당 정책 가능** — 단, m2net 가 직접 지원하는 게 아니라 **간접 구현**
- m2net 는 그냥 종량제로 계속 과금 → **우리 앱(오리진하우스)이 시간 도달 시 종료 호출** = "건당" 처럼 동작

### 단가 등록 방식
- 예: "15분당 10,000원" 정책 = **10,000원 / 910초** 로 등록
  - 15분 = 900초이지만 약간 여유 두기 위해 910초
  - m2net 가 종량제로 차감 시 정확히 10,000원이 빠지도록

### 우리(사주플랜)의 역할
- **900초 도달 시 채팅 종료 호출** → 자동 종료

### 두 정책 혼용 불가
- **같은 상담사가 종량제 + 건당 동시 X**
- 상담사별로 **한 가지 정책만** 선택 가능
- → 운영 정책 결정 필요:
  - 옵션 A: 모든 상담사 건당으로 일괄 전환
  - 옵션 B: 상담사가 자유롭게 선택 (시스템 복잡도 ↑)
  - 옵션 C: 신규 상담사부터 건당, 기존은 종량제 유지

## 3. 연장 흐름 (사장님 질문 — 자동화 가능?)

**결론: 100% 자동화 가능. m2net 변경 0.**

### 원리
m2net 는 계속 종량제로 과금하므로, **우리 앱이 종료 호출만 안 하면 그게 곧 연장**.

### UX 흐름 (15분 / 10,000원 기준)
```
t=0    채팅 시작 → 사용자 결제 (10,000원 = 코인 차감)
       m2net 종량제 차감 시작
t=720s 사용자에게 "곧 종료됩니다. 연장하시겠어요?" 모달 (3분 전 알림)
        ├─ [연장] → 추가 결제 (10,000원)
        │   - 우리 앱은 종료 호출 안 함
        │   - m2net 계속 과금 (누적 1,820초까지)
        │   - t=1620s 에 다시 3분 전 알림
        │   - t=1810s 종료 호출 (실제 30분)
        └─ [종료] 또는 [무응답]
            - t=900s 종료 호출
            - 사용자에게 "15분 사용 완료" 안내
```

### 핵심 포인트
- 사용자가 연장 결제 시 → 단순히 "타이머 +900초" 만 늘림
- m2net 입장: 그냥 종량제 채팅이 길어진 것
- 모든 사이클이 **우리 앱 내부 로직**으로 동작

## 4. 개발 분담

| 작업 항목 | 담당 | 비고 |
|---|---|---|
| 단가 등록 (10,000원/910초 형식) | m2net | 가맹점 페이지 수정 |
| 별도 메시지 코드/플래그 | m2net (가능성) | 종량제와 구분이 필요한지 m2net 와 추가 확인 |
| 시작 시 결제 처리 | 사주플랜 (백엔드) | 10,000원 차감 + 채팅 세션 시작 |
| 900초 카운터 | 사주플랜 (백엔드 또는 프론트) | 백엔드 권장(시계 보안) |
| 종료 호출 (m2net API) | 사주플랜 (백엔드) | 기존 채팅 종료 로직 활용 |
| 3분 전 연장 알림 모달 | 사주플랜 (프론트) | 사용자 UX |
| 연장 결제 흐름 | 사주플랜 (프론트+백엔드) | 같은 채팅 세션 유지 |
| "건당/종량제" 상담사 정책 분기 | 사주플랜 (백엔드+프론트) | 운영 정책 결정 후 |
| 통계 / 매출 분류 (건당 vs 종량) | 사주플랜 (백엔드) | 영업이익 시뮬레이터 연동 |

## 5. m2net 에게 추가 확인 필요한 점

m2net 메일에 명시 안 됨 → 확인 필요:

1. **시간 도달 알림** — m2net 가 우리 앱에 "910초 됐어요" 알림 push 주나? 아니면 우리가 자체 타이머로 카운트?
2. **종료 API 동일?** — 건당 종료 호출은 기존 종량제 종료와 같은 API 인가, 별도인가
3. **부분 환불 처리** — 사용자가 5분만 쓰고 끄면 → 환불 가능? 아니면 그대로 10,000원 차감?
4. **별도 결제 코드?** — m2net 콜백/푸시에서 "이건 건당이다" 구분되나, 그냥 종량제와 동일?
5. **연장 시 재결제 흐름** — 같은 세션 유지하면서 추가 차감 가능한가, 새 세션 만들어야 하는가
6. **CPID 추가?** — 건당 / 종량제 별도 CPID 발급 필요한가, 같은 CPID 안에서 단가 옵션으로 가능한가

## 6. 진행 순서 (의존성)

```
[1] 사주플랜 운영 정책 결정 (옵션 A/B/C)
        ↓
[2] m2net 에 "건당 정책으로 가겠다" 공식 통보 + Section 5 질문 송부
        ↓
[3] m2net 내부 개발 (단가 등록, 콜백 코드 등) ← 사주플랜 대기
        ↓
[4] m2net 완료 통보
        ↓
[5] 사주플랜 개발 시작
     - 백엔드: 결제 흐름, 900초 카운터, 종료 호출, 연장 결제
     - 프론트: 연장 알림 모달, 종료 안내, UX
        ↓
[6] 통합 테스트 (TEST 환경 sajumoon.kr 에서)
        ↓
[7] PROD 배포 (sajuplan.com)
```

**중요**: [3] 동안 사주플랜은 **대기 X** — 백엔드/프론트 UI 작업은 m2net 의존성 없이 진행 가능.
구체적으로:
- 900초 카운터, 종료 호출 로직, 연장 모달 UI 등은 **mock m2net 응답으로 단위 테스트 가능**
- 실제 m2net 연동은 [4] 이후 통합 테스트에서만
