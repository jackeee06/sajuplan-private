## 후불 시스템 설계 (개발자 설명 기반 — 2026-06-08)

### 과금 구조 (개발자 설명)

```
10분 통화 예시 (고객 단가: 30초당 1,000원 기준):

고객 청구액: (600초 ÷ 30초) × 1,000원 = 20,000원
상담사 수익: 20,000원 × 정산률 (등급별 40~70%)
m2net 인프라: (600초 ÷ 10초) × 18원 = 1,080원
결제수수료: 고객 청구액 × 3.5% (결제 방식에 따라 다름)
사주플랜 몫: 나머지
```

- **과금 단위**: 선불과 동일하게 30초 단위 사용 권장 (사용자 혼란 방지)
- **고객 화면**: 통화 종료 후 총 상담료 표시 → 결제 요청
- **상담사 화면**: 기존과 동일 (30초 단위 수익금 적립)

## 후불 결제 방식 — 미결정 (3가지 옵션)

### 옵션 A: 060 핸드폰 요금 청구 (m2net 060 서비스)
- m2net과 별도 060 계약 필요: 월 +30만원 (현재 70만원 → 100만원)
- 060 번호 발급 필요 (공개 번호, 고객이 직접 다이얼)
- 통신사(KT/SKT/LGU+)가 수납액의 10% 수수료 차감
- 미수금 발생 가능 (못 받는 돈이 다음달로 이월)
- **장점**: 고객이 별도 결제 행동 불필요
- **단점**: 10% 수수료 높음, 미수금 리스크

### 옵션 B: 등록된 카드 자동 청구 (사주플랜 자체 구현)
- 통화 종료 후 고객의 등록 카드로 자동 결제
- AG9 BillKey(자동충전과 동일 방식) 활용 가능
- 카드 미등록 고객은 후불 사용 불가
- **장점**: 수수료 3.5%로 낮음, 미수금 없음
- **단점**: 카드 등록 사전 필요, 개발 공수 큼

### 옵션 C: 앱 내 결제 요청 (인앱 결제 화면)
- 통화 종료 후 "N원 결제하기" 화면 노출
- 고객이 직접 결제 (코인 충전처럼 AG9 PG 결제)
- **장점**: 기존 PG 인프라 재사용
- **단점**: 고객이 결제를 안 하면 미수금 발생, UX 불편

## 후불 활성화를 위해 해야 할 것 (순서대로)

### Step 1: 결제 방식 결정 (사장님)

- 이 결정 없이 개발 진행 불가

### Step 2: m2net 협의 (결제 방식에 따라)

- 옵션 B/C 선택 시: m2net에 후불 번호 설정 협의

### Step 3: 개발

- `setting.consult.phone_postpaid` → 실제 060 번호로 교체
- `POSTPAID_ENABLED = true` 로 변경

#### 옵션 B 추가 개발

#### 옵션 C 추가 개발
